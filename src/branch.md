`fetch` 是一个新命令，它基本上与 `push` 相反。
`push` 将书签和提交从本地仓库发送到远程，而 `fetch` 则下载其他人（比如 Bob）可能从另一台计算机推送的书签和提交。

这是我们以前没见过的。
我们的版本历史被分成两个**分支**。
这并不罕见，也是 Jujutsu 使人们能够彼此独立工作的核心所在。
但我们还不知道如何处理这种情况。

如果 Jujutsu 允许 Alice 在不知道 Bob 更新的情况下推送 `main` 书签，他的工作就会被意外删除。
记住：远程仓库只认为那些从书签**可达**的提交值得保留。
但如果 `main` 书签指向 Alice 的提交，那么 Bob 的提交就不再是**可达的**了，因为它不是 Alice 提交的祖先。

注意，`main` 书签在日志中**出现了两次**，两次都带有问号 **`??`**。
这意味着 Jujutsu 不确定书签应该指向哪里。
Alice 将其移到了她的提交上，而在远程上它被移到了 Bob 的提交上。
一旦 Alice 决定了它实际应该指向哪里，她就可以通过显式地将其移动到那里来告诉 Jujutsu。

```admonish info title="令人困惑的术语：书签和分支"
术语"书签"和"分支"经常在非常相似的情况下使用，这可能会令人困惑。
让我们非常精确地定义它们的区别，并准备好识别潜在的混淆来源。

**书签**只是一个命名的标签，它附着于或指向一个单独的提交。
就是这样。

一个**分支**由其**尖端**来指定，尖端也是一个单独的提交。
然而，一个分支指的是**一组提交**，包括尖端及其**所有祖先**。

这两个概念可以在像这样的短语中结合起来：
"我将我的更改推送到了 `main` 分支。"
这可能会令人困惑，因为 `main` 是一个书签，对吗？
为什么我们在谈论"main"分支？
嗯，`main` 书签指向一个提交，而那个提交就是我们说"main 分支"时所指的分支的尖端。

在与主要使用 Git 的人交谈时，还有一件事需要注意。
Git 总是使用 **branch** 这个词来表示这两种东西。
"bookmark" 这个术语在 Git 中没有意义。
所以，当 Git 用户说 "branch" 这个词时，他们可能是在谈论一个实际的分支，即一组包含尖端及其所有祖先的提交。
他们也可能是在谈论一个书签，即一个指向单个提交的命名标签，而丝毫不关心该提交的祖先。
```
