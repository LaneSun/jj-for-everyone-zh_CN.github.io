Alice 和 Bob 刚刚参加了下一堂编程课。
除了其他内容，他们终于学习了 Python 中的循环。
Alice 现在想回到她的实验中来修复它：

```sh
jj new 'description(substring:"WIP：添加 for 循环")'
```

她的代码目前看起来像这样：

```py
for (i = 0; i < 10; i = i + 1):
    print("Hello, world!")
```

这个语法是错误的，所以 Alice 用她在讲座中学到的语法来修复它：

```py
for _ in range(10):
    print("Hello, world!")
```

您可以手动编辑文件，也可以运行以下命令：

```sh
echo 'for _ in range(10):
    print("Hello, world!")' > hello.py
```

```sh
jj commit -m "修复循环语法"
```

您可能已经猜到这是如何发生的。
合并的一方，`main` 分支，以一种方式更改了文件 `hello.py`（添加了更多语言）。
另一方添加了循环。
Jujutsu 不知道如何组合这些更改。
这种情况被称为**冲突**。

## 解读冲突标记

在文本编辑器中打开文件 `hello.py`，或运行 `cat hello.py` 来查看冲突文件的内容：

```
<<<<<<< conflict 1 of 1
%%%%%%% diff from: krymmwqm 1581674f "合并 hello-world 的代码和文档"
\\\\\\\        to: quvtvrzl d0c3efcd "Print German and French greetings as well"
 print("Hello, world!")
+print("Hallo, Welt!")
+print("Bonjour, le monde!")
+++++++ oxnuoryz c6042124 "修复循环语法"
for _ in range(10):
    print("Hello, world!")
>>>>>>> conflict 1 of 1 ends
```

哎呀，这看起来很复杂。
让我们试着理清我们所看到的内容，以便逐步理解冲突。

关注第一行和最后一行：

```
<<<<<<< Conflict 1 of 1
...
>>>>>>> Conflict 1 of 1 ends
```

这两行表示冲突的开始和结束。
在较大的文件中，通常文件的大部分可以无冲突地合并。
只有当合并的双方修改相同的行时，Jujutsu 才会报告冲突。
这些冲突行的块然后表现为带编号的部分，带有类似上面的开始和结束标记。
在我们的示例中，文件太小了，以至于整个过程就是一个冲突。

接下来，关注冲突内的第一部分：

```
%%%%%%% diff from: krymmwqm 1581674f "合并 hello-world 的代码和文档"
\\\\\\\        to: quvtvrzl d0c3efcd "Print German and French greetings as well"
 print("Hello, world!")
+print("Hallo, Welt!")
+print("Bonjour, le monde!")
```

冲突标记告诉我们这些更改来自哪些范围的提交。
第一部分向我们展示了在 `main` 分支上对冲突行所做的更改。
显示更改的格式是常见的"diff"格式。
在这种格式中，所有行都以空格、减号 `-` 或加号 `+` 为前缀。
空格表示该行未被更改。
减号表示该行被删除，加号表示该行被添加。
被_更改_的行表示为一行删除和一行添加。

这种情况下，我们有一行未更改（英语问候）和两行已添加（德语和法语的问候）。

现在来看冲突的第二部分：

```
+++++++ oxnuoryz c6042124 "修复循环语法"
for _ in range(10):
    print("Hello, world!")
```

这部分没有向我们显示某个提交范围内的更改，它只是向我们展示了描述为"修复循环语法"的提交的最终状态。
原因是之前的状态可以从第一部分推断出来。

这个解释可能显得过于冗长。
但合并冲突可能会比这更复杂。
希望从现在开始，您面对它们时会稍微更有准备一些。

## 修复冲突

既然我们理解了冲突各方的状态，我们如何修复它？
这里**没有简单的解决方案**！
如果有的话，Jujutsu 早就帮您做了。
合并冲突的关键在于正确的解决方案取决于冲突更改的含义。

在这种情况下，冲突的更改是：
- 在循环中打印问候语十次。
- 也打印德语和法语的问候语。

合并这两项更改时，德语和法语的问候语也应该重复十次吗？
如果所有语言都应重复，应该交错还是整齐分开？

举例来说，这些更改有三种不同的合理合并方式：

1. 只重复英语问候语：
    ```py
    for _ in range(10):
        print("Hello, world!")
    print("Hallo, Welt!")
    print("Bonjour, le monde!")
    ```

1. 重复所有语言并交错排列：
    ```py
    for _ in range(10):
        print("Hello, world!")
        print("Hallo, Welt!")
        print("Bonjour, le monde!")
    ```

1. 重复所有语言但分开排列：
    ```py
    for _ in range(10):
        print("Hello, world!")
    for _ in range(10):
        print("Hallo, Welt!")
    for _ in range(10):
        print("Bonjour, le monde!")
    ```

我想您明白这一点的含义了。
为了正确解决合并冲突，您通常需要动动脑筋。

让我们选择第二个选项作为我们的冲突解决方案。
您可以手动编辑冲突标记，或者直接用以下命令替换文件：

```sh
echo 'for _ in range(10):
    print("Hello, world!")
    print("Hallo, Welt!")
    print("Bonjour, le monde!")' > hello.py
```

冲突解决本身被视为一种更改，所以该提交不像我们创建的第一个合并提交那样被标记为"(empty)"。
让我们提交并推送到 main：

```sh
jj commit -m "合并问候语的重复和翻译"
jj bookmark move main --to @-
jj git push
```

```admonish info title="变基期间的冲突"
冲突也可能在执行变基时发生。
不过那些解决起来稍微更难一些。
我们将在下一等级中学习它。
如果您在执行变基时遇到冲突，暂时考虑放弃（使用 `jj undo`）而改为创建一个合并提交。
```

````admonish note title="有助于冲突解决的工具" collapsible=true
上面我教您的是通过将冲突标记替换为期望的结果来手动解决冲突。
这种手动方法总是有效的。
不过，手动解决冲突可能会很繁琐。

有各种工具可以在特定情况下帮助您解决冲突。
例如，内置工具 `:ours` 和 `:theirs` 让您丢弃冲突的一方，只保留另一方。
哪一方是"ours"，哪一方是"theirs"可能并不明显。
只需尝试其中一个，如果需要的话运行 `jj undo`。
例如：
```sh
# 保留"我们"的更改，丢弃"他们"的更改
jj resolve --tool :ours
```
除了这些简单的内置工具外，Jujutsu 还可以调用更强大的外部工具。
其中一个例子是 [Mergiraf](https://mergiraf.org/)。
它尝试基于冲突文件的编程语言的**语法树**自动修复冲突。
不过，Jujutsu 并不自带 Mergiraf，如果您想使用它，需要单独安装。
安装完成后，可以这样使用：
```sh
jj resolve --tool mergiraf
```
这之所以可行，是因为 Jujutsu 有一个关于如何调用 Mergiraf 的默认工具配置。
可能还有其他 Jujutsu 不知道的工具。
如果您想使用其中一个，可以添加自定义的[合并工具配置](https://docs.jj-vcs.dev/latest/config/#3-way-merge-tools-for-conflict-resolution)。
````
