Alice 给自己买了一台新笔记本电脑，她对此非常兴奋。
这是一台 [Framework 笔记本电脑](https://frame.work/)，因为 Alice 已经厌倦了计算机制造商故意让她难以修理自己的硬件。
Framework 也官方支持 Linux，这对 Alice 来说非常重要。
尊重用户的硬件加上尊重用户的软件是绝佳的组合，Alice 这样认为。

无论如何，重点是：Alice 不再拥有她的仓库了。
她本可以从旧电脑上复制过来，但既然她可以直接重新克隆仓库，她就没有这样做。
让我们模拟 Alice 重新克隆仓库：

```sh

rm -rf ~/jj-tutorial/repo
jj git clone ~/jj-tutorial/remote ~/jj-tutorial/repo
cd ~/jj-tutorial/repo

# 扮演 Alice
jj config set --repo user.name "Alice"
jj config set --repo user.email "alice@local"
jj metaedit --update-author
```

指向循环实验提交的书签显示为 `push-zkwmvzlqluot@origin`。
但是 `@origin` 是怎么回事？
以前没有这个。
它意味着该书签只存在于远程"origin"上，本地没有。
我们可以通过_跟踪_该书签来解决这个问题，但让我们详细讨论一下这意味着什么。

## 书签跟踪

Jujutsu 有一个概念，即书签可以是**已跟踪**或**未跟踪**的。
如果书签被跟踪，Jujutsu 将在您运行 `jj git push` 和 `jj git fetch` 时尝试保持其本地和远程版本同步。
这种区分有几个原因。
例如，您可能创建书签以便更容易地引用它们所指向的提交，而不打算将提交推送到任何远程。
另一方面，远程可能存储着您本地不关心的书签，例如您团队成员的工作进展。
在这两种情况下，拥有书签都是合理的。
但您不希望它们通过 `jj git push` 和 `jj git fetch` 在本地和远程仓库之间传输。

到目前为止，我们不必担心书签的"跟踪状态"。
我们使用了两个不同的命令将新书签推送到远程：
- `jj git push --bookmark <NAME>`
- `jj git push --change <REVISION>`

这两个命令都会自动将书签设置为跟踪状态，因为这显然是用户想要的。
然而，还有一个命令 `jj git push`，可以同时推送多个书签。
使用该命令时，Jujutsu 将拒绝推送之前未跟踪过的书签。

相反的情况是，书签存在于远程，但本地没有跟踪书签。
默认情况下，`jj log` 的输出不会显示仅在远程存在的书签上的提交，因为用户可能对它们不感兴趣。
但 `push-zkwmvzlqluot` 书签不是别人的书签，而是 Alice 的。
她希望该书签被跟踪，这样她可以在日志中默认看到它，并且以后可以更新和推送它。
跟踪书签的命令是：

```sh
jj bookmark track push-zkwmvzlqluot
```
