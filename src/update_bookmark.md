# 更新书签

````admonish reset title="重置您的进度" collapsible=true
要将您的进度重置到本章开头，请运行以下命令：

```sh
curl https://jj-for-everyone.github.io/reset.sh | bash -s update_bookmark
cd ~/jj-tutorial/repo
```
````

我们已经学会了如何首次将新书签推送到远程。
我们_可以_对每个想要创建并推送到远程的提交都这样做。
然而，这会导致大量不必要的书签散落在各处。
真是一团糟！

谢天谢地，有更好的方法。
我们不必每次都创建新书签，而是可以**移动**现有书签，使其指向另一个提交。

要试一试，我们首先需要创建一个新提交。
您可以直接复制这些命令：

```sh
printf "\nThis is a toy repository for learning Jujutsu.\n" >> README.md
jj commit -m "Add project description to readme"
```

上面的命令使用了一个巧妙的技巧。
如果您已经想好了简短的描述，可以使用 `-m` 标志（`--message` 的缩写）直接将其传递给 `jj commit`。
这比打开单独的文本编辑器更快。

`jj log` 向我们展示了在推送到远程的提交之上新增的提交：

现在我们可以先将书签指向这个新提交，然后再次推送书签，从而将这个新提交发送到远程。
让我们从移动书签开始：

```sh
jj bookmark move main --to @-
```

我们本可以使用变更ID 告诉 Jujutsu 将书签移动到哪里，即 `--to kxqyrwux`。
或者，我们可以使用 `@-`，这是引用**工作副本提交的父提交**的一种巧妙方式。
有很多这样巧妙的方式来引用提交，但那是以后的话题。
现在，只需记住 `@-`，因为这是目前最有用的一个。

让我们看看 `jj log` 说了什么：

这里需要注意的重要事项是，Jujutsu 向我们展示了本地书签 `main` 与其远程对应物之间的差异。
`main*` 是本地书签，名称旁边的星号表示它与远程不同步。
`main@origin` 是书签在远程上的位置。

我们可以通过再次推送书签来修复这种情况：

```sh
jj git push
```

这次我们没有明确指定 `--bookmark main`。
如果在 `push` 命令上省略 `--bookmark` 标志，Jujutsu 会尝试智能地判断您实际想要推送哪些书签。
这通常效果很好。
如果您认为它没有起作用，只需加上 `--bookmark` 标志再试一次。

您可能想知道：
既然远程需要书签来接收提交，而 `main` 书签**不再**指向第一个提交了……那个提交现在丢失或被删除了吗？
幸运的是，没有。
提交存储了对其父提交的引用，这就是为什么 Jujutsu 知道在 `jj log` 输出中绘制提交的顺序。
这些引用形成了一条链，保护被标记提交的所有祖先免于丢失。
